# 数据湖集成 — Kafka 数据格式说明

本文档供数据湖厂家集成对接，说明推送至 Kafka 的消息格式、字段含义及消费方式。

## 1. Kafka 连接信息

| 项目         | 说明                               |
|-------------|------------------------------------|
| **Brokers** | `10.81.151.147:21007`, `10.81.151.148:21007`, `10.81.151.149:21007` |
| **Topic**   | `gyhlw_dp_test`                    |
| **认证方式** | Kerberos / GSSAPI (SASL)           |
| **序列化**  | Key 和 Value 均为纯文本 JSON (UTF-8) |

## 2. 消息语义

MQTT 侧接收到的是一条包含多个测点的批量消息，服务端将其**拆分为逐测点单条**写入 Kafka：

- **每 1 条 Kafka 消息 = 1 个测点的 1 次采样**
- **消息 Key** = 测点名 (`tag_name`)，确保同一测点写入同一分区，保证该测点数据的时序有序
- **消息 Value** = 下方 JSON

## 3. Kafka Value 格式

### 3.1 JSON 示例

```json
{
  "tag_name": "DDM.SIS.1DCS_10DCS_TDM_01",
  "device_id": "sis-collect-dev-dy",
  "value": 26.5,
  "timestamp": 1600397414,
  "quality": 1
}
```

### 3.2 字段说明

| 字段        | 类型    | 必填 | 说明                                                      |
|------------|--------|------|----------------------------------------------------------|
| `tag_name` | string | 是   | 测点名称，全局唯一。也是这条消息的 Kafka Key                 |
| `device_id`| string | 是   | 设备标识，标识数据来源设备                                  |
| `value`    | double | 是   | 测点数值                                                    |
| `timestamp`| int64  | 是   | 采集时间戳，**Unix 秒级**（如 `1600397414` = 2020-09-18 02:50:14 UTC） |
| `quality`  | int    | 是   | 质量码：`1` = 良好 (Good)，`0` = 异常 (Bad)                |


## 4. Kafka Message Key

- **格式**：测点名 (`tag_name`) 的 UTF-8 字节
- **作用**：利用 Kafka 的 Hash 分区策略，同一测点始终路由到同一分区，保证消费时该测点数据严格有序
- **消费端无需特殊处理**，直接读取即可；如业务上需要按设备聚合，可按 `device_id` 二次分组

## 5. 消息示例（多条连续记录）

以下是 Kafka Topic `gyhlw_dp_test` 中可能连续出现的 3 条消息：

```json
{"tag_name":"DDM.SIS.1DCS_T0101","device_id":"sis-collect-dev-dy","value":23.1,"timestamp":1600397002,"quality":1}
{"tag_name":"DDM.SIS.1DCS_T0102","device_id":"sis-collect-dev-dy","value":45.7,"timestamp":1600397002,"quality":1}
{"tag_name":"DDM.SIS.1DCS_T0103","device_id":"sis-collect-dev-dy","value":12.0,"timestamp":1600397003,"quality":0}
```

## 6. 消费注意事项

1. **编码**：Key 和 Value 均为 UTF-8 字符串，Value 为合法 JSON，可直接 `JSON.parse()` 解析
2. **时序**：同一 `tag_name` 的消息保证分区内有序；不同测点之间不保证全局时序
3. **去重**：本服务使用 `At-Least-Once` 语义（QoS 1），极端情况下可能存在重复消息，建议消费端按 `tag_name + timestamp` 做幂等去重
4. **数据密度**：消息频率取决于现场设备采集频率，高峰期每秒可达数百条
5. **质量码**：`quality=0` 的记录仍会写入 Kafka，消费端可根据业务需求过滤或标记
